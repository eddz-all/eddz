## 第 2 页：项目背景与痛点

标题：

为什么需要 ProjectPilot

核心内容：

- 多人协作时，项目可能分布在多台服务器上。
- 不同服务器的 Git 状态、Python/Node/Docker 环境经常不一致。
- 传统 SSH 批量登录执行命令存在安全风险。
- 操作过程缺少统一记录，出问题后难追踪。
- 只靠人工排查成本高，容易遗漏环境差异和 Git 风险。

讲述重点：

这个项目不是单纯展示页面，而是解决“多服务器项目状态看不清、执行不安全、过程不可追踪”的问题。

## 第 4 页：总体架构

标题：

总体架构：中央控制 + 分布式执行

页面内容：

使用框图展示：

```text
用户
  ↓
前端控制台
  ↓ HTTP API
ProjectPilot 总后端
  ↓
数据库

ProjectPilot 总后端
  ↑ HTTPS 轮询 / 回传
各服务器上的 Executor
  ↓
本机 Git / 环境检测 / 命令执行
```

核心说明：

- 前端只和总后端通信。
- Executor 也只和总后端通信。
- 前端不直接连接服务器。
- 总后端统一创建任务、保存结果、记录日志。
- Executor 主动轮询任务，执行后回传结果。

讲述重点：

这个架构避免了后端大量 SSH 登录服务器，改为目标服务器主动连接总后端，安全边界更清楚。

## 第 8 页：Executor 模式

标题：

Executor：服务器主动拉取任务

核心内容：

- 每台目标服务器运行一个 Executor Agent。
- Executor 定期请求总后端：
  - `POST /executor/poll`
- 如果有任务，后端返回任务内容。
- Executor 在本机执行检测或命令。
- 执行完成后回传结果：
  - `POST /executor/tasks/{task_id}/result`
- 后端保存结果并更新状态。

状态流转：

```text
queued -> running -> completed
queued -> running -> failed
```

图示建议：

画一个时序图：

```text
前端点击检测
  ↓
后端创建 ExecutorTask
  ↓
Executor poll
  ↓
Executor 本机执行
  ↓
Executor result
  ↓
后端写入状态和日志
```

技术亮点：

- 不需要后端主动 SSH 到每台服务器。
- 目标服务器只需能访问总后端 HTTPS 地址。
- 支持异步任务状态展示。
- 任务和执行结果全程可追踪。

## 第 9 页：Git 与环境检测能力

标题：

检测能力：看清项目在每台服务器上的真实状态

核心内容：

- Git 检测：
  - 当前分支
  - remote 信息
  - ahead / behind
  - 是否有未提交改动
  - 最近提交
  - 是否为 Git 仓库
- 环境检测：
  - 操作系统
  - CPU 架构
  - Python 版本
  - Node.js 版本
  - Docker 是否安装和运行
  - CUDA 版本
  - 磁盘占用

重要改进：

- 后端综合状态接口会返回最新成功快照。
- 如果最新检测失败，不再用旧 Git 快照误导前端。
- 新增 `latest_git_detection` 和 `latest_environment_detection` 展示最近检测任务状态。

讲述重点：

系统不只看“以前检测成功过什么”，还会关注“最近一次检测是否失败”，避免前端展示过期状态。

## 第 11 页：安全执行与日志追踪

标题：

安全机制：AI 生成，人工确认，后端拦截

核心内容：

- 执行前需要用户确认。
- 后端对命令做安全检查。
- 危险命令会被阻断。
- 每个任务都有状态：
  - `queued`
  - `running`
  - `completed`
  - `failed`
- 所有关键行为写入 OperationLog。

可讲的安全点：

- 不让前端直接操作服务器。
- 不让 AI 直接无确认执行命令。
- Executor 只执行分配给自己的任务。
- Executor 限制 `allowed-root`，只能访问允许目录。
- 后端记录任务输入、执行结果和错误信息。

讲述重点：

这个项目强调“受控自动化”，不是黑盒自动执行。
